# Полный запуск проекта локально (русская инструкция)

Документ для очень подробного старта с нуля, особенно если вы раньше почти не работали с Docker.

---

## 1) Что нужно понимать перед стартом

В проекте есть 3 основные части:

1. **Frontend** — то, что вы видите в браузере.
2. **Backend** — сервер, который обрабатывает запросы frontend.
3. **PostgreSQL** — база данных, в которой всё хранится.

Обычно всё это нужно настраивать отдельно, но Docker позволяет запускать проект в контейнерах, где уже подготовлено нужное окружение.

То есть ваша задача как новичка сводится примерно к такому сценарию:
- установить Docker,
- открыть терминал,
- перейти в папку проекта,
- выполнить одну команду запуска.

---

## 2) Что нужно установить

### Обязательно для простого запуска
- **Git** — чтобы скачать репозиторий.
- **Docker Desktop** (Windows/macOS) или **Docker Engine + Docker Compose plugin** (Linux).

### Нужно только для запуска без Docker
- **Node.js 20+**
- **npm 10+**
- **PostgreSQL 16+**

---

## 3) Как проверить, что всё установилось

Откройте терминал и выполните:

```bash
git --version
docker --version
docker compose version
```

Если команды выводят версии, а не ошибки, значит всё установлено правильно.

Если `docker` не находится:
- Docker не установлен,
- либо установлен, но не запущен,
- либо терминал был открыт до установки и его нужно закрыть/открыть снова.

## 4) Как скачать проект

Если репозиторий ещё не скачан:

```bash
git clone <URL_ВАШЕГО_РЕПО>
cd Site
```

Если папка уже есть, просто перейдите в неё:

```bash
cd Site
```

Важно запускать команды именно из корня проекта — из папки, где лежит `docker-compose.yml`.

Проверить это можно так:


## 5) Рекомендуемый запуск (через Docker)

Это **самый простой и безопасный путь** для новичка.

### Шаг 1. Убедиться, что Docker запущен

#### Windows / macOS
Откройте **Docker Desktop** и дождитесь, пока программа покажет, что Docker работает.

#### Linux
Проверьте:

```bash
docker info
```

---

### Шаг 2. Запустить сборку и контейнеры

Выполните в корне проекта:

```bash
docker compose up --build
```

Что будет происходить:
- Docker прочитает `docker-compose.yml`.
- Будет поднят контейнер `db` с PostgreSQL.
- Будет собран и запущен контейнер `api`.
- Будет собран и запущен контейнер `web`.
- Backend дождётся готовности БД.
- Затем применятся миграции и загрузятся тестовые данные.

Если запуск первый, не пугайтесь, что это долго: Docker может скачивать образы из интернета и собирать слои впервые.

---

### Шаг 3. Открыть проект в браузере

После успешного старта откройте:

- **Сайт:** `http://localhost`
- **API:** `http://localhost:8080`

База данных будет доступна на:
- **PostgreSQL:** `localhost:5432`

Тестовые учётные данные:
- `admin / admin123`
- `curator / curator123`
- `student / student123`

---

### Шаг 4. Как понять, что всё работает

Обычно признаки успеха такие:
- в терминале нет постоянного потока ошибок;
- команда не завершилась аварийно;
- `http://localhost` открывается в браузере;
- можно войти под тестовым пользователем.

Дополнительно можно проверить состояние контейнеров:

```bash
docker compose ps
```

Если сервисы поднялись, вы увидите контейнеры `db`, `api`, `web`.

---

### Шаг 5. Как остановить проект

Если команда запущена в текущем терминале, нажмите:

```bash
Ctrl + C
```

Чтобы затем корректно остановить контейнеры и сеть проекта, выполните:

```bash
docker compose down
```

Если нужно **полностью сбросить базу данных** и поднять всё «с нуля»:

```bash
docker compose down -v
```

## 6) Как запускать проект повторно

После первого успешного запуска обычно используется одна из двух команд:

### В обычном режиме с логами в текущем окне

```bash
docker compose up
```

### В фоновом режиме

```bash
docker compose up -d
```

Если вы меняли код или конфигурацию Docker, используйте:

```bash
docker compose up --build
```

Или в фоне:

```bash
docker compose up -d --build
```

## 7) Что делает каждая часть Docker-конфигурации

### `docker-compose.yml`
Это главный файл orchestration-конфигурации. Он говорит Docker:
- какие сервисы запускать,
- какие порты открыть,
- какие переменные окружения передать,
- в каком порядке запускать сервисы.

### Сервис `db`
Поднимает PostgreSQL 16.

### Сервис `api`
Запускает backend и подключает его к базе данных.

### Сервис `web`
Запускает frontend, который доступен в браузере.

### Volume `pgdata`
Это отдельное постоянное хранилище для PostgreSQL.
Благодаря ему данные не исчезают просто от остановки контейнера.

## 8) Запуск одной командой через predeploy-скрипт

Если вы хотите сначала проверить проект, а потом запустить Docker, используйте скрипт:

```bash
chmod +x scripts/predeploy.sh
./scripts/predeploy.sh
```

Что делает скрипт:
1. проверяет наличие Docker;
2. проверяет `docker compose`;
3. проверяет наличие lockfile;
4. запускает проверки проекта;
5. проверяет корректность `docker-compose.yml`;
6. поднимает контейнеры.

Это удобно, если вы хотите запускать проект в более «проверенном» режиме.

### Все логи сразу

```bash
docker compose logs
```

### Логи только backend

```bash
docker compose logs api
```

### Логи только frontend

```bash
docker compose logs web
```

### Логи только PostgreSQL

```bash
docker compose logs db
```

### Логи в режиме «следить вживую»

```bash
docker compose logs -f
```

Остановить просмотр логов можно через `Ctrl + C`.

## 10) Полный сброс, если «всё поехало»

Иногда проще полностью очистить локальное окружение проекта и поднять всё заново:

```bash
docker compose down -v
docker compose up --build
```

Это полезно, если:
- сломались данные в БД,
- завис старый контейнер,
- вы не уверены, в каком сейчас состоянии окружение.

## 11) Типичные проблемы и решения

### Проблема: команда `docker compose up --build` не работает
Проверьте:
- установлен ли Docker,
- запущен ли Docker Desktop,
- находитесь ли вы в папке проекта,
- существует ли файл `docker-compose.yml`.

### Проблема: ошибка `port is already allocated`
Это значит, что нужный порт уже занят другой программой.
Чаще всего конфликтуют:
- `80`
- `8080`
- `5432`

Что делать:
- закрыть программу, которая использует порт,
- или изменить проброс порта в `docker-compose.yml`.

### Проблема: сайт открывается, но ничего не работает
Проверьте backend:

```bash
docker compose logs api
```

И статус контейнеров:

```bash
docker compose ps
```

### Проблема: база не стартует
Проверьте:

```bash
docker compose logs db
```

Если у вас уже установлен локальный PostgreSQL, он может занимать порт `5432`.

### Проблема: после перезапуска всё пропало
Возможно, вы запускали:

```bash
docker compose down -v
```

Ключ `-v` удаляет volume базы данных, а значит стирает локальные данные PostgreSQL для этого проекта.

## 12) Запуск без Docker (если очень нужно)

Этот путь сложнее, но иногда полезен для разработки.

### Шаг 1. Настроить backend

```bash
cd server
cp .env.example .env
npm install
npm run migrate
npm run seed
npm run dev
```

Что делает `.env`:
- хранит настройки порта,
- параметры подключения к PostgreSQL,
- JWT secrets.

Пример значений есть в `server/.env.example`.

### Шаг 2. Запустить frontend в другом терминале

```bash
cd ..
npm install
npm run dev
```

### Шаг 3. Убедиться, что PostgreSQL у вас установлен и запущен

Для ручного режима без Docker база должна быть поднята отдельно.

## 15) Короткая памятка для совсем быстрого старта

Если вам нужен самый короткий путь, то последовательность такая:

1. Установить Docker.
2. Открыть Docker Desktop.
3. Скачать репозиторий.
4. Перейти в папку проекта.
5. Выполнить:

```bash
docker compose up --build
```

6. Открыть `http://localhost`.
7. Для остановки выполнить:

```bash
docker compose down
```
