# Стратегия отдельных PR (чтобы не ловить конфликты)

## Зачем
Когда изменения большие, GitHub чаще блокирует merge из-за конфликтов и сложнее пройти проверку CI.

## Рекомендуемый формат
- 1 PR = 1 задача (например: только backend auth, только секции, только Docker).
- Размер PR: до 15–20 файлов, без смешивания инфраструктуры и бизнес-логики.
- Перед началом работы: `git fetch origin && git rebase origin/main`.

## Обязательный чеклист перед PR
1. `npm ci`
2. `npm run check:contracts`
3. `npm run check:store`
4. `npm run build`

## Если появились конфликты
1. Обновить ветку: `git fetch origin && git rebase origin/main`
2. Разрешить конфликтные файлы вручную.
3. Повторить чеклист.
4. Отправить: `git push --force-with-lease` (после rebase).

## Рекомендация для диплома
Лучше делать серию маленьких PR (backend → frontend → deploy → docs), чем один большой PR.
